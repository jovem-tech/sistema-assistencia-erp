# Release v2.7.0 - Migracao legada SQL

Data: 28/03/2026
Tipo: MINOR

## Objetivo
Adicionar uma infraestrutura segura e auditavel para importar `clientes`, `equipamentos` e `ordens de servico` de um sistema antigo para o ERP novo.

## O que mudou

### 1. Infraestrutura de banco e rastreabilidade
- `clientes`, `equipamentos` e `os` passaram a aceitar:
  - `legacy_origem`
  - `legacy_id`
- `os` passou a gravar tambem:
  - `numero_os_legado`
- foram criados indices unicos por origem/ID legado para reprocessamento sem duplicidade
- foi criado indice de busca para `numero_os_legado`

### 2. Conexao secundaria para o legado
- o sistema agora possui um grupo `legacy` em `app/Config/Database.php`
- a conexao e parametrizada por `.env` com `database.legacy.*`

### 3. Pipeline de migracao
- novo arquivo de configuracao: `app/Config/LegacyImport.php`
- novo servico central: `app/Services/LegacyImportService.php`
- novos auxiliares:
  - `LegacyRecordNormalizer`
  - `LegacyCatalogResolver`

Sequencia implementada:
1. importar clientes
2. importar equipamentos
3. importar OS

### 4. Comandos publicos
- `php spark legacy:preflight`
- `php spark legacy:import --execute`
- `php spark legacy:report`

### 5. Auditoria da carga
- nova tabela `legacy_import_runs`
- nova tabela `legacy_import_events`

Essas tabelas registram:
- execucao
- status final
- totais por entidade
- erros
- avisos
- conflitos ignorados

### 6. Integracao da OS com referencia legada
- a listagem `/os` agora aceita busca por `numero_os_legado`
- a coluna `N OS` pode exibir:
  - numero oficial
  - numero legado
  - origem
- a visualizacao `/os/visualizar/{id}` mostra `Numero legado` e `Origem` quando houver importacao

## Regras operacionais
- o numero oficial do ERP continua em `numero_os`
- o numero antigo fica apenas como referencia em `numero_os_legado`
- o pipeline nao faz mesclagem heuristica por nome
- status legado sem mapeamento bloqueia a carga
- catalogos podem ser criados automaticamente quando habilitado
